# 게시판 다중 선택 삭제, 이동, 복사 Design

## 1. 목표

게시판 목록에서 다중 선택한 글을 삭제, 이동, 복사할 수 있게 한다.  
작업 후에는 반드시 GET 목록 화면으로 복귀하고, 새로고침 시 POST 재실행이 발생하지 않도록 한다.

## 2. 핵심 설계 원칙

- POST 처리와 화면 복귀는 분리한다.
- 목록 화면은 현재 게시판의 단일 진입점만 사용한다.
- form action과 redirect는 같은 기준 URL을 사용한다.
- 작업 결과 메시지는 flash 세션으로 넘긴다.
- 브라우저 뒤로가기나 새로고침이 POST를 재생성하지 않도록 PRG 패턴을 강제한다.

## 3. 요청 흐름

### 3.1 목록 진입

1. 사용자는 게시판 목록을 연다.
2. 목록 스킨은 게시판별 체크박스와 bulk toolbar를 렌더링한다.
3. toolbar의 form action은 목록의 표준 진입 URL을 사용한다.

### 3.2 bulk 제출

1. 사용자가 `선택삭제`, `이동`, `복사`를 누른다.
2. JS는 선택 항목, 대상 게시판, 액션 유효성을 1차 점검한다.
3. 서버는 CSRF, 권한, 대상 게시판, 선택 글 존재 여부를 재검증한다.
4. 서버는 실제 작업을 수행한다.
5. 성공 또는 실패 메시지를 flash에 저장한다.
6. 서버는 GET 목록 화면으로 redirect 한다.

### 3.3 복귀

1. 브라우저는 GET 목록 화면을 다시 요청한다.
2. 목록 화면은 flash 메시지를 읽어 1회만 표시한다.
3. 새로고침 시에도 GET만 재요청되므로 POST가 반복되지 않는다.

## 4. URL 정책

- 목록 form action은 게시판의 표준 목록 URL을 사용한다.
- redirect 역시 동일한 목록 URL 규칙을 사용한다.
- 닷홈 환경에서 `pretty URL`과 실제 파일 경로가 다를 수 있으므로, 게시판 bulk 기능은 반드시 실제 동작 검증이 끝난 단일 URL 규칙만 사용한다.
- POST 이후에는 URL에 `board`, `q`, `page`만 남기고 나머지는 제거한다.

## 5. 서버 검증 규칙

### 5.1 공통

- CSRF 필수
- 로그인 필요
- 게시판 관리 권한 필요
- 선택한 글이 하나 이상 존재해야 함

### 5.2 삭제

- 같은 게시판 안에서만 삭제
- 첨부파일, 댓글, 에디터 이미지까지 함께 정리

### 5.3 이동

- 대상 게시판은 현재 게시판과 달라야 함
- 대상 게시판 쓰기 권한 필요
- 첨부파일 레코드와 본문 내 에디터 이미지 경로를 대상 게시판으로 재배치

### 5.4 복사

- 대상 게시판 쓰기 권한 필요
- 원본 글은 유지
- 새 글을 생성하고 첨부파일도 복제
- 본문 내 에디터 이미지는 복제된 파일 경로로 치환

## 6. UI 설계

- bulk toolbar는 목록 상단에 공통 배치한다.
- toolbar에는 `전체 선택`, `대상 게시판 선택`, `선택삭제`, `이동`, `복사`를 둔다.
- 체크박스는 각 게시글 항목에만 둔다.
- toolbar와 체크박스는 공통 JS로 연결한다.
- 유효성 오류는 브라우저 alert로 1차 처리하고, 서버 오류는 alert 영역으로 표시한다.

## 7. 데이터 영향

- 새 데이터 구조 추가는 없다.
- 기존 `board_posts`, `board_comments`, `board_files`를 사용한다.
- bulk 로그는 기존 audit 로그에 남긴다.

## 8. 예외 처리

- 선택 글이 없으면 작업하지 않는다.
- 대상 게시판이 없으면 작업하지 않는다.
- 권한이 없으면 작업하지 않는다.
- 작업 중 일부 글이 실패하면 전체 작업을 중단하고 실패 메시지를 반환한다.
- 실패 시에도 redirect 흐름은 유지해 POST 재전송을 막는다.

## 9. 테스트 시나리오

1. 관리자 계정으로 삭제 후 목록 복귀 확인
2. 관리자 계정으로 이동 후 원본/대상 게시판 상태 확인
3. 관리자 계정으로 복사 후 원본 유지 및 새 글 생성 확인
4. 선택 없이 제출 시 방어 확인
5. 대상 게시판 미선택 시 방어 확인
6. 일반 계정에서 toolbar 미노출 확인
7. 새로고침 시 POST 재실행 여부 확인
8. 닷홈 배포 환경에서 404 재발 여부 확인

## 10. 구현 메모

- redirect 경로는 한 곳에서만 생성한다.
- form action도 한 곳에서만 생성한다.
- 메시지는 flash에 넣고 목록 GET에서 읽는다.
- 이전페이지 새로고침으로 POST가 다시 실행되지 않도록 PRG를 엄격히 유지한다.
