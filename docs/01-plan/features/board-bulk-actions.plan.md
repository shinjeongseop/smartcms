# 게시판 다중 선택 삭제, 이동, 복사 PDCA Plan

## 1. 개요

게시판 목록에서 다중 선택으로 글을 삭제, 이동, 복사할 수 있도록 구현한다.  
현재 기능은 삭제/이동/복사 처리 자체는 동작하지만, 처리 직후 404가 발생하고 새로고침 시에도 POST가 재실행되어 부작용이 생긴다.

## 2. 문제 정의

- 선택 삭제 후 404 Not Found가 발생한다.
- 새로고침 시 이전 POST가 반복 실행되어 대상 게시판 선택 오류가 다시 노출된다.
- 실제 데이터는 삭제되거나 이동, 복사되지만 사용자 경험이 깨진다.
- 현재 라우팅, form action, redirect, flash 메시지 흐름이 서버 환경과 완전히 맞지 않을 가능성이 높다.

## 3. 목표

- 선택 삭제, 이동, 복사 기능을 안정적으로 제공한다.
- 작업 후에는 항상 안전한 GET 목록 화면으로 복귀한다.
- 새로고침해도 POST가 반복 실행되지 않도록 한다.
- 닷홈 환경에서도 404 없이 동작하도록 라우팅을 단순화한다.

## 4. 범위

### 포함

- 게시판 목록 bulk action 폼 구조 점검
- POST 처리 후 redirect 경로 점검
- 삭제, 이동, 복사 후 PRG(Post-Redirect-Get) 보장
- 새로고침 재전송 방지
- flash 메시지 전달 검증

### 제외

- 게시판 스킨 전면 개편
- bulk UI 디자인 재작업
- 별도 권한 모델 재설계
- 첨부파일, 썸네일, 에디터 기능 재구현

## 5. 현재 가설

1. 폼 action 또는 redirect가 닷홈 라우팅과 충돌하고 있다.
2. redirect가 성공하지 못해 404 페이지로 떨어진다.
3. 브라우저가 POST를 유지한 상태에서 새로고침되며 재전송 경고 또는 재처리가 발생한다.
4. GET 목록 화면으로 확실히 복귀하지 못해 flash 메시지 표시 타이밍이 꼬인다.

## 6. 성공 기준

- 삭제, 이동, 복사 후 404가 발생하지 않는다.
- 작업 완료 후 목록 화면으로 정상 복귀한다.
- 새로고침 시 POST가 재실행되지 않는다.
- 결과 메시지가 목록 화면에서 한 번만 표시된다.
- 선택한 글이 없을 때, 대상 게시판이 없을 때, 권한이 없을 때 각각의 오류가 명확하게 분리된다.

## 7. 검증 항목

1. 선택 삭제
2. 선택 이동
3. 선택 복사
4. 선택 없음
5. 대상 게시판 없음
6. 권한 부족
7. 관리자 계정과 일반 계정의 분기
8. 새로고침 및 뒤로가기 동작

## 8. 리스크

- 닷홈의 rewrite 규칙이 파일 경로와 pretty URL을 다르게 취급할 수 있다.
- form action과 redirect가 서로 다른 경로 체계를 쓰면 다시 404가 날 수 있다.
- flash 메시지가 redirect 이전에 소실될 수 있다.
- bulk 처리 중 일부 글만 성공하는 부분 성공 상태가 생길 수 있다.

## 9. 다음 단계

1. 게시판 bulk action의 실제 요청 경로와 redirect 경로를 분리해서 점검한다.
2. POST 후에는 반드시 GET 목록 화면으로만 돌아가도록 설계를 확정한다.
3. 닷홈 환경에서 가장 안전한 URL 패턴을 design 문서에 명시한다.
4. 검증 후 코드 수정으로 이어간다.
